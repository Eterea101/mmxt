2.2 YAML frontmatter

必须填写的 元数据

每个 SKILL.md 文件开头都有一段 YAML frontmatter，用于定义 Skill 的元数据：

---
name: my-skill # Skill的唯一标识符
description: | # 描述：什么时候该用这个Skill
Helps write product requirement documents.
Use when user mentions PRD, feature spec,
or requirements.
---

必须字段

name 使用 kebab-case 格式，无空格无大写，如 "prd-writer"
description 清晰描述这个 Skill 做什么，以及什么时候该使用它

术语表

类比

专业解释

3.1 什么是渐进式披露

按需 逐步展开

渐进式披露是 Skill 设计中的核心原则：不要在开头堆砌所有细节，而是先给出核心概念，按需展开。

核心优势

这种设计可以 Token 消耗降低高达 90%，同时保持完整的功能性。

"就像一位优秀的老师，不会第一天就把所有知识塞给学生，而是根据学生的进度逐步深入。"

组织 指令的框架

一个好的 Skill 指令应该按照这个顺序组织：

渐进式披露的四个层次

关键是 How 部分要放在最后，并且可以使用折叠或条件触发的方式，让它在真正需要的时候才出现。

第三章 3.3/5
3.3 触发条件的艺术

避免 两个极端

description 中的触发短语决定了 Skill 是否会被加载。太模糊会导致 undertriggering（该加载时不加载），太宽泛会导致 overtriggering（不该加载时却加载）。

两种常见错误

Undertriggering Skill应该加载却不加载 - 触发短语太少或太具体
Overtriggering Skill不该加载却加载了 - 触发短语太泛

"描述技能时，使用主动语态和具体的触发短语，能让Claude更准确地判断何时该调用这个Skill。"

什么是主动语态 + 具体的触发短语？

好的触发短语示例：

"帮我写 PRD"

"创建一个产品需求文档"

"需要写功能规格说明"

不好的触发短语示例：

"帮忙"（太模糊，什么都可以触发）

"写文档"（太泛，任何文档都可能触发）

主动语态 vs 被动语态

主动语态 "Use when user says 'write a PRD'" — 明确告知AI何时行动
被动语态 "This skill might be used for documentation" — AI无法判断何时该用

类比

专业解释
Skill 应该被加载却没有被加载的问题。发生原因是触发短语太少或太具体。
类比
就像每次说"帮我个忙"手机都响，但其实你只是想聊天。
专业解释
Skill 不该被加载却被加载了的问题。发生原因是触发短语太宽泛。
类比

专业解释

Create 模式

从零开始 构建 Skill

什么是 Create 模式

Create 是 skill-creator 的核心功能，通过引导式问答，帮助你从零构建一个完整的 Skill。

工作原理

Claude 会根据你的回答自动生成 SKILL.md 的内容，包括：

description 字段的触发短语

Instructions 核心指令

Examples 示例

"你描述得越具体，产出的 Skill 就越精准。整个过程约5-10分钟。"

"描述很具体" vs "描述模糊"

模糊的描述：

"我想创建一个帮助写文档的 skill。"

具体的描述：

"我想创建一个专门写 PRD（产品需求文档）的 skill。它应该：

1. 帮助用户按标准模板撰写 PRD

2. 在用户提到'PRD'、'产品需求文档'时触发

3. 输出包含问题陈述、成功指标、优先级

4. 不擅长：技术设计、竞品分析"

第二个问题是：用户什么时候应该调用这个 Skill？

触发短语应该：

具体且相关

包含同义词

覆盖常见表达

"当用户提到 PRD、产品需求文档、功能规格、需求说明时，都应该触发这个 Skill。"

好的触发短语（具体且相关）

触发短语：

"帮我整理会议纪要"

"这个会议说了什么"

"提取会议的行动项"

"会议要点总结"

不好的触发短语（宽泛且无关）

"帮忙" → 任何场景都可能触发

"文档" → 任何文档相关都触发

"写东西" → 完全无法判断意图

正反对比

好 "帮我整理会议纪要" — 明确指向任务
差 "帮我个忙" — 不知道帮什么忙
好 "提取行动项" — 明确提取动作
差 "处理任务" — 任何任务都触发
